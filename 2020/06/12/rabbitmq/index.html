rabbitmq新手入门学习

rabbitmq官网

参考博客

1.MQ概念

消息队列（Message Queue，简称MQ），从字面意思上看，本质是个队列，FIFO先入先出，只不过队列中存放的内容是message而已。
其主要用途：不同进程Process/线程Thread之间通信

为什么会产生消息队列？有几个原因：

  • 不同进程（process）之间传递消息时，两个进程之间耦合程度过高，改动一个进程，引发必须修改另一个进程，为了隔离这两个进程，在两进程间抽离出一层（一个模块），所有两进程之间传递的消息，都必须通过消息队列来传递，单独修改某一个进程，不会影响另一个；
  • 不同进程（process）之间传递消息时，为了实现标准化，将消息的格式规范化了，并且，某一个进程接受的消息太多，一下子无法处理完，并且也有先后顺序，必须对收到的消息进行排队，因此诞生了事实上的消息队列；

    2.rabbitmq简介

    rabbitmq是使用Erlang （面向并发的编程语言）开发的一个开源的，基于AMQP（Advanced Message Queueing Protocol）协议的完整的可复用的消息系统。,支持主流操作系统Windows，linu，Macos，多种开发语言支持。

    3.安装

    对于windows平台官网提供了两种安装方法：
  • 第一种是最优方法，使用chocolatey（windows平台下的一种包管理器）安装，直接在powershell或者cmd输入一条命令就行,前提是安装chocolatey并且配好环境。
    choco install rabbitmq

4.学习五种队列


代码博客里面都有，我只说说我的理解

  • 第一种：简单队列

sender将消息发送到队列，receier从队列获取，sender和receiver彼此独立

  • 第二种：work模式

一个生产者，多个消费者。自动确认时是采用轮询方式，也就是不管消费者处理任务的速度如何，消息总是轮流给消费者处理。举个例子，给小孩子分糖吃，大盘子里的糖果由生产者生产，每个小孩前面都有一个自己的小盘子，你将糖轮流一个个放到小孩子的盘子里，小孩子从自己的盘子里拿糖吃，小孩子吃糖有快有慢，但是现在小孩子吃糖和分糖是彼此独立的，因为小盘子的存在。但是小盘子这种数据结构是否存在我并不知道，但是我感觉这样才说得通。

如果一个消费者在处理一个消息时被杀死了，他处理的消息也就会没了，这显然不是理想的。而且如果我想让消费者能者多得而不是像上面那样公平分配，吃糖吃的快的可以多吃。那就要手动确认，只有当一个消费者处理完消息发送确认，队列收到确认后才能继续获取消息。这样当一个消费者被杀死，由于没有发送确认，这里是队列重新发送还是生产者重新发送呢，讲道理按照目前的认知这还不是很清楚，因为队列里的消息被获取后除非有备份，对哦，应该是队列重新发送，消费者获取消息时，队列并不会像上面简单队列那样直接pop，而是将队尾消息复制发送给消费者，只有当队列收到消费者的确认时才pop。然后，能者多劳，就像上面的吃糖例子，现在不是轮流发到小盘子里，而是吃完了才能继续吃。小孩子拿到糖吃完后告诉你我吃完了，你听到了才继续发。

  • 第三种：publish/subscribe模式

果然跟我之前的想法是一致的，生产者并不是直接发送消息到队列，而是发送到了交换机。其实生产者并不和队列直接打交道，他们之间有着一个媒介–交换机（exchange），交换机并没有存储消息的能力，只有队列有。每个消费者都有自己的队列，那么交换机怎么知道将消息转发到哪个队列，那就要将交换机与队列就行绑定，进行绑定了的消费者队列就相当于订阅了该生产者。这里也就解释了我之前的疑惑，交换机就相当于装糖的大盘子，但这个大盘子实际上不是存储糖的，而是分糖的中转站，由他负责将糖分给订阅了他的小孩子，每个小孩子前的小盘子也就是每个消费者的队列，对分到的糖进行存储，小孩子从小盘子里拿糖。

这里如果先运行消费者，会报vhost里找不到声明的exchange，因为exchange并没有创建，那个博主的代码里消费者只声明了队列，并没有声明exchange，exchangeDeclare()和queueDeclare()方法如果exchange和queue不存在的话就创建。

exchange有多种类型：direct，headers，topic和fanout，fanout也就是广播，将消息转发到所有绑定的消费者队列，一个消费者队列也可以有多个消费者，但是每一次只有一个消费者获取到，那么每次是哪个消费者获取呢，是随机还是轮流呢，我试了一下，结果是轮流。如果两个消费者声明了同一个队列，那么这两个消费者轮流接收消息。

  • 第四种：路由模式

fanout将消息广播给所有订阅的消费者，那么如果有些消息不想要让一些消费者收到呢，这就要用路由模式。首先将exchange的类型改为direct，然后basicPublish的第二个参数传入routingkey，消费者绑定队列时通过第三个参数与routingkey绑定。既然是路由模式，我们可以用路由器来理解（看作交换机也可以，反正是多端口就行），将exchange看作一个多端口的路由器，每个端口对应一个routingkey，比如上图中就有error，info，warning三个端口，现在消费者通过绑定连接到路由器的一个或者多个端口，生产者生产消息时会提交消息对应的端口，路由器接收到消息后将消息转发到指定端口的队列。比如error的消息通过error端口转发出去，由于c1和c2两个消费者绑定该端口，所有两个队列都会收到。

  • 第五种：topic主题模式

路由模式只通过一个规则进行匹配，那么如果我想匹配多个规则呢？举个例子，上面通过问题的严重性error，warning，info来匹配，现在我想添加一个颜色，红色的error该给哪个消费者队列，绿色的info又该给哪个。显然路由模式并不满足我们的需求，那就要用到topic模式，也叫适配器模式。*只能匹配单个word，#可以匹配0个或多个words。
